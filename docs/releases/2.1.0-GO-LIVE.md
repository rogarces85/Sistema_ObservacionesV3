## Alcance del release

Auditoria y endurecimiento del sistema actual. No se introducen
funcionalidades nuevas visibles para el usuario final, salvo:

- Modales de confirmacion consistentes con el diseno del sistema
  (reemplazan dialogos nativos del navegador).
- Reset de contrasena via email cuando SMTP esta configurado;
  fallback a visualizacion al supervisor si no.
- Trazabilidad reforzada: cada cambio queda registrado en
  `historial_estados` o `historial_usuarios` con timestamp,
  autor y, donde aplique, motivo.

No se modifico el modelo de datos, no se agregaron dependencias
nuevas, no se cambio el diseno visual general. Es 100%
compatible con la base de datos de 2.0.0.

## Lo que se ejecuta fuera del repositorio

El equipo de operaciones debe ejecutar, en el orden indicado
en `deploy/README.md`:

1. `bash deploy/setup-server.sh` (B1) en servidor Ubuntu 22.04.
2. Clonar `main` en `/var/www/rem` y correr `composer install`.
3. `mysql < deploy/sql/create-db-user.sql` (B4).
4. Crear `/etc/rem/env.php` desde `deploy/env.production.example`.
5. `bash deploy/migrate.sh` (B5).
6. `php deploy/create-first-supervisor.php --username=admin ...` (B6).
7. `php deploy/purge-demo-users.php --keep=admin` (B7).
8. `cp deploy/apache-rem.conf /etc/apache2/sites-available/rem.conf` y
   `a2ensite` (B9).
9. `bash deploy/setup-https.sh rem.example.cl` (B3).
10. Activar `rem-worker.service` y `rem-worker.timer` (B10).
11. `cp deploy/fail2ban-* /etc/fail2ban/` y reiniciar (B12).
12. `cp deploy/backup.sh /etc/cron.daily/rem-backup` (B11).
13. `cp deploy/healthcheck.sh` y configurar cron cada 5 min (B14).
14. Smoke test final desde la URL publica.

## Plan de rollback

Si 2.1.0 falla en produccion dentro de los 7 dias posteriores
al go-live:

```bash
# Rollback de codigo
cd /var/www/rem
sudo -u www-data git fetch --tags
sudo -u www-data git checkout v2.0.0
sudo systemctl reload apache2

# Si fue una falla de BD (snapshot v002, etc.) restaurar backup
sudo bash deploy/restore.sh YYYY-MM-DD HHMMSS
```

El tag `v2.0.0` se creara con el commit `a4ece6d` para que el
rollback sea deterministico.

## Riesgos conocidos

- **SMTP no configurado al inicio**: el reset de contrasena
  funcionara via fallback (supervisor ve la contrasena). El
  equipo de operaciones debe configurar SMTP antes del primer
  reset real, o documentar que el supervisor debe comunicarla
  por canal seguro.
- **CDN de Tabler**: la UI usa `cdn.jsdelivr.net` para Tabler,
  Chart.js y tabler-icons. Si la red bloquea el CDN en el
  entorno corporativo, la UI no cargara. Plan B: hostear
  localmente (no esta en este release).
- **Datos de prueba residuales**: si el dump de la BD oficial
  actual se usa como base, pueden quedar `admin123` en cuentas
  demo. `purge-demo-users.php --keep=admin` los desactiva.
- **Cambio de password en primer login**: no implementado. Los
  usuarios demo quedan con `admin123` hasta que se les cambie
  manualmente. Documentado en `faltantes.md` como deuda tecnica.

## Decision

Pasamos a go-live. El sistema esta listo para ser desplegado en
el entorno de produccion siguiendo `deploy/README.md`. El equipo
de operaciones es responsable de ejecutar el despliegue fisico
y el cambio de DNS, con acompanamiento del equipo de desarrollo
durante las primeras 24 horas.

Si surge algun incidente durante el go-live:

1. Activar `deploy/healthcheck.sh` para alertas automaticas.
2. Revisar `OPERATIONS.md` seccion "Incidentes comunes".
3. Si es critico, ejecutar rollback a `v2.0.0`.
4. Reportar al canal de soporte y abrir ticket.
