## Cambios por categoria

### Seguridad

- **CSRF obligatorio** en endpoints mutables:
  - `api/users.php` (POST/PUT/DELETE)
  - `api/assignments.php` (POST)
  - `api/deleted.php` (POST)
  - `api/observations.php` (ya estaba, sin cambios funcionales)
  - `api/import.php` (solo en `confirm`, no en `preview`)
  - `api/report_queue.php` (POST)
  - `api/versioning.php` (POST)
  - Implementacion: `includes/csrf.php::validateRequest()` con
    busqueda en `X-CSRF-TOKEN` header o `csrf_token` body.

- **Confirmaciones explicitas en backend** (no solo en UI):
  - `api/users.php?action=reset_password` requiere `confirm_reset: true`.
  - `api/users.php?action=delete` (DELETE) requiere `confirm_delete: true`.
  - `api/deleted.php?action=permanent_delete` y `permanent_delete_multiple`
    requieren `confirm_irreversible: true`.
  - `api/versioning.php?action=rollback` mantiene doble confirmacion
    (modal + tipeo de "ACEPTAR" en UI).

- **Guards contra auto-acciones de supervisor** (en `api/users.php`):
  - No puede cambiar su propio rol (`rol` distinto al actual).
  - No puede desactivarse a si mismo (`activo=0` con `userId == id`).
  - No puede eliminarse a si mismo (DELETE con `userId == id`).
  - No puede resetear su propia contrasena via admin (redirigido a perfil).

- **Validaciones de input endurecidas**:
  - `api/assignments.php`: anio 2020-(anio+1), meses 1-12 con dedupe y
    clamp, IDs enteros positivos, tipos `anual|temporal`.
  - `api/assignments.php`: `copiar_anio` rechaza origen == destino.
  - `api/users.php`: normalizacion de inputs, helper
    `requireExistingUser`, `requireExplicitConfirmation`.

- **Reset de contrasena aleatorio** (`api/users.php`,
  `includes/Mailer.php`):
  - `Mailer::generateRandomPassword(12)` produce 12 chars con al
    menos una mayuscula, minuscula, digito y simbolo.
  - `api/users.php?action=reset_password` ahora genera password
    aleatorio, intenta enviar via `Mailer`, y si SMTP no esta
    configurado, devuelve el password al supervisor con `warning`
    explicito en la respuesta.
  - `includes/Mailer.php` soporta SMTP TLS/SSL con AUTH LOGIN.
    Fallback: escribe en `/var/log/rem/mail.log` (o tmp).

- **Headers HTTP endurecidos** (en `deploy/apache-rem.conf`):
  - `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`
  - `Content-Security-Policy: default-src 'self'; ...`
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy: geolocation=(), microphone=(), camera=()`
  - Bloqueo de `config/`, `includes/`, `models/` en Apache.
  - `php_flag engine off` en `uploads/` con `AddType text/plain`.

- **Config endurecida** (`config/config.php`):
  - `display_errors = 0` en produccion.
  - `log_errors = 1` con ruta configurable.
  - `session.cookie_secure = 1` por defecto en produccion.
  - `session.cookie_samesite = 'Lax'`.
  - `session.use_strict_mode = 1`.
  - Credenciales de BD solo desde `REM_ENV_FILE` o env vars;
    sin defaults inseguros hardcoded.
  - `config/.env.local.php` (gitignored) permite dev local sin
    setup manual.

### UI / feedback

- **Modales consistentes** (`assets/js/confirm-modal.js`):
  - `remConfirm({ title, message, confirmText, danger, requireText })`
  - `remPrompt({ title, message, label, requireText, multiline })`
  - `remAlert({ title, message, variant })`
  - Cargado globalmente desde `includes/footer.php`.
  - Reemplaza `confirm()` y `prompt()` nativos en:
    `views/usuarios.php` (delete + reset),
    `views/asignaciones.php` (remover + copiar + reasignacion),

    `views/observaciones.php` (delete a papelera con motivo).

- **Bloqueo de doble submit** en:
  - `views/supervision.php` (Aprobar/Cancelar/Eliminar masivos)
  - `views/observaciones.php` (Guardar, Importar confirmar)
  - `views/eliminadas.php` (Restaurar/Eliminar masivo)
  - `views/usuarios.php` (Crear/Editar, Toggle estado, Eliminar,
    Reset password)

  - `views/reportes.php` (Encolar Excel/PDF)

  - Soft delete via `DeletedObservation::moveToTrash` con motivo.

  - `confirm` requiere CSRF; `preview` no muta.

  - POST/PUT/DELETE con CSRF.
  - `password` policy: 8+ chars, mayuscula, digito.
  - `reset_password` con `confirm_reset` y password aleatorio.
  - `update` no permite cambiar rol propio.

- **`api/assignments.php`**:
  - POST con CSRF.
  - Helpers `normalizeAssignmentYear`, `normalizeMeses`.
  - `asignar`, `asignar_multiple`, `remover`, `copiar_anio`.

- **`api/deleted.php`**:
  - POST con CSRF.
  - `permanent_delete*` requiere `confirm_irreversible`.
  - `restore` valida ID existente.

- **`api/locations.php`**:
  - POST con CSRF en `create`/`update`/`toggle`.

- **`api/report_queue.php`**:

## Procedimiento de rollback a 2.0.0

Si la version 2.1.0 falla en produccion:

```bash
# En el servidor de produccion
cd /var/www/rem
sudo -u www-data git fetch
sudo -u www-data git checkout a4ece6d   # ultimo commit de 2.0.0
sudo -u www-data composer install --no-dev --optimize-autoloader
sudo systemctl reload apache2
```

La migracion a 2.0.0 puede requerir restore desde backup
si se crearon datos en formato 2.1.0 (por ejemplo, snapshot
v002). Coordinar con DBA antes de rollback.

## Procedimiento de upgrade de 2.0.0 a 2.1.0

```bash
# En el servidor de produccion
cd /var/www/rem
sudo -u www-data git pull origin main
sudo -u www-data composer install --no-dev --optimize-autoloader
sudo systemctl reload apache2
```

Sin cambios de schema; sin migraciones SQL nuevas. Las
modificaciones de `config/config.php` requieren:

1. Crear `/etc/rem/env.php` con la estructura de
   `deploy/env.production.example` o `config/config.production.example`.
2. Setear `SetEnv REM_ENV_FILE /etc/rem/env.php` en el VirtualHost.
3. Si no se hace, el sistema devuelve 500 con instruccion clara.
